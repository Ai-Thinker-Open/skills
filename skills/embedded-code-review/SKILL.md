# 安信可嵌入式 C 代码深度审查

对嵌入式 C 代码进行多维度深度审查。审查时加载 `ai-thinker-c-coding-standard` 规范作为基础，再叠加本 skill 的深层审查规则。

## 审查流程

1. **加载编码规范** — 先读取 `ai-thinker-c-coding-standard`，逐项核对基础规范
2. **分层审查** — 按本 skill 的维度逐层深入
3. **输出报告** — 按严重程度分级（🔴 致命 / 🟠 严重 / 🟡 警告 / 🔵 建议），每条给出位置、问题描述、修正代码

---

## 审查维度

### 1. 安全审查

#### 1.1 输入校验
- 所有函数入口是否对参数做合法性检查（空指针、范围、边界）
- 外部输入（UART/SPI/I2C/网络数据）是否做长度/格式校验后再使用
- 用户可控的长度字段是否用于缓冲区操作前做了上界检查
- 数组索引是否保证在有效范围内

#### 1.2 缓冲区安全
- 是否存在缓冲区溢出（`strcpy`/`sprintf`/`memcpy` 无长度限制）
- 栈上大数组（>256B）在嵌入式环境是否有栈溢出风险
- 环形缓冲区读写指针是否有并发保护
- DMA 缓冲区对齐是否满足硬件要求

#### 1.3 整数安全
- 有符号/无符号混合运算是否可能导致隐式转换问题
- 乘法/加法是否有溢出风险（尤其在计算缓冲区大小时）
- 位移操作是否超出类型宽度
- 枚举值是否有未覆盖的分支

#### 1.4 指针安全
- 指针解引用前是否检查空指针
- `malloc` 返回值是否检查（如必须使用动态分配）
- 指针算术是否保证在合法内存范围内
- 函数指针调用前是否验证有效性

#### 1.5 并发安全
- 共享变量是否用 `volatile` 修饰（ISR 与主循环共享）
- 临界区保护是否完整（禁中断范围是否最小化）
- 自旋锁/互斥量使用是否正确，是否可能导致死锁
- ISR 中是否做了耗时操作（阻塞、长循环、动态分配）
- 多核环境下缓存一致性问题

---

### 2. 内存管理

#### 2.1 动态内存（应尽量避免）
- 是否有不必要的 `pvPortMalloc`/`malloc` 调用（嵌入式优先静态分配）
- 分配大小是否受控，避免用户输入直接决定分配量
- 是否存在内存泄漏（分配后无对应释放路径）
- 是否存在重复释放
- 释放后是否将指针置 `NULL`（防止野指针）
- 是否有碎片化风险（频繁小块分配释放）

#### 2.2 栈内存
- 递归调用是否有深度限制
- 函数栈帧大小是否合理（局部大数组、大结构体）
- FreeRTOS 任务栈大小是否足够（通过 `uxTaskGetStackHighWaterMark` 验证）
- 中断栈是否预留足够空间

#### 2.3 静态内存
- 全局变量是否真的需要全局作用域
- 大型缓冲区是否放在 `static` 而非栈上
- `const` 数据是否放在 Flash 区（`const` 修饰）
- 外设寄存器映射是否使用 `volatile`

---

### 3. FreeRTOS 任务规范

#### 3.1 任务创建
- 任务优先级是否合理（避免优先级反转、优先级饥饿）
- 任务栈大小是否经过测量验证（`configMINIMAL_STACK_SIZE` 作为基线）
- 任务入口函数签名是否正确：`void vTaskFunc(void *pvParameters)`
- 任务是否设置了合理的超时或心跳检测
- 空闲任务是否被阻塞（不应在空闲任务中做耗时操作）

#### 3.2 任务间通信
- 队列使用是否有超时机制（避免永久阻塞）
- 信号量/互斥量是否有配对的 give/take
- 事件组是否处理了超时和错误情况
- 通知机制是否比队列更高效（轻量场景优先用任务通知）

#### 3.3 任务同步
- 临界区（`taskENTER_CRITICAL`/`taskEXIT_CRITICAL`）是否最短
- 中断屏蔽时间是否可控
- 是否优先使用 `FromISR` 版本的 API（ISR 中不得调用非 FromISR 版本）
- 任务调度是否被意外禁止（`vTaskSuspendAll` 未配对 `xTaskResumeAll`）

#### 3.4 内存与资源
- 任务删除（`vTaskDelete`）后是否释放其栈内存
- 每个任务是否有独立的错误处理路径
- 任务是否在退出前清理所有分配的资源
- 软件定时器回调中是否做了耗时操作（应在任务中处理）

---

### 4. 中断与硬件安全

#### 4.1 ISR 规范
- ISR 是否尽量短小（只做必要的标志设置和数据搬运）
- ISR 中是否避免了阻塞操作（`vTaskDelay`、等待队列、等待信号量）
- ISR 中是否避免了动态内存分配
- ISR 是否使用 `FromISR` 版本的 API
- 中断嵌套是否安全（ISR 中是否正确使能/禁止中断）
- ISR 与主循环共享的数据是否用 `volatile` + 适当同步保护

#### 4.2 外设安全
- GPIO 配置是否考虑了上电默认状态
- 外设初始化顺序是否正确（时钟→引脚→外设）
- UART/SPI/I2C 通信是否有超时机制
- DMA 传输完成后是否正确清理
- ADC 采样是否考虑了参考电压稳定性

#### 4.3 看门狗
- 看门狗是否在关键操作前喂狗
- 看门狗超时设置是否大于最长阻塞路径
- 是否有明确的看门狗喂狗策略

---

### 5. 编码规范合规（基于 ai-thinker-c-coding-standard）

#### 5.1 基础规范核对
- [ ] 统一前缀（`axk`/`aiio`/`ai`）是否在同一模块内一致
- [ ] `.h` 对外接口是否有 Doxygen 函数头注释（`@brief`/`@param`/`@return`）
- [ ] 函数头注释是否英文，首字母大写，句号结尾
- [ ] 函数命名是否按层级（适配层 `<pfx>_组件_功能`，应用层 `动词_名词`）
- [ ] 全局变量 `g_<pfx>_`，静态变量 `s_<pfx>_`
- [ ] 宏 `<PFX>_` 全大写，含多表达式的宏参数和整体加括号
- [ ] 结构体/枚举标签 `<pfx>_` 前缀，typedef 后 `_t` 结尾
- [ ] 4 空格缩进，禁止 Tab
- [ ] `if`/`for`/`while` 即使一行也加 `{}`
- [ ] 函数参数超过 5 个用结构体封装
- [ ] `.h` 加防重复包含宏 `#ifndef <PFX>_XXX_H`
- [ ] 函数功能单一，不超过 200 行

---

### 6. 可维护性与健壮性

#### 6.1 错误处理
- 函数返回值是否统一（成功/失败枚举）
- 错误路径是否有资源清理（goto 清理模式）
- 是否有 assert 用于调试阶段的前置条件检查
- 关键操作是否有重试机制

#### 6.2 可移植性
- 是否使用了编译器特定扩展（如 `__attribute__`）需要条件编译
- 数据类型是否使用固定宽度类型（`int32_t`、`uint8_t` 而非 `int`）
- 位操作是否考虑了大小端
- 平台相关代码是否隔离在适配层

#### 6.3 可读性
- 魔数是否定义为宏或常量
- 条件表达式是否简洁清晰（避免过深嵌套）
- 命名是否见名知意
- 注释是否解释 WHY 而非 WHAT

## 审查原则

1. **安全第一** — 致命和严重问题必须修复，警告问题应尽快修复
2. **逐项核对** — 不跳过任何维度，即使某维度"应该没问题"
3. **给出修正** — 每个问题都提供具体的修正代码或方向
4. **上下文理解** — 理解代码意图后再审查，避免误报
5. **嵌入式思维** — 考虑资源受限环境（RAM/Flash 有限、无异常机制、实时性要求）
